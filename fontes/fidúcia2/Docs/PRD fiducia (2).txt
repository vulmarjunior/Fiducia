0. Instruções para a IA Desenvolvedora
Este documento é o contrato de desenvolvimento entre o dono do sistema e a IA responsável pelas implementações. Antes de qualquer alteração de código, leia integralmente este PRD.
0.1 Stack e convenções obrigatórias
* Frontend: React 18+ com Vite. Componentização em /src/components/<Modulo>/<Componente>.jsx.
* Banco de dados: Google Cloud Firestore (SDK v9 modular). Nunca usar SDK compat/legado.
* Autenticação: Firebase Auth — Google Sign-In. Nunca remover ou contornar.
* Estilo: CSS Modules ou Tailwind (manter o padrão existente no projeto). Não introduzir novos sistemas de estilo.
* Testes: Cada nova função utilitária deve ter pelo menos 1 teste Jest correspondente.
* Sem dependências desnecessárias: Avaliar bundle size antes de adicionar npm packages.
0.2 Regras de comportamento
* Nunca apagar funcionalidades existentes sem instrução explícita.
* Nunca alterar regras de segurança do Firestore sem aprovação explícita.
* Commits atômicos: Uma funcionalidade por PR, com mensagem descritiva em português.
* Toda UI nova deve suportar modo claro e escuro (CSS variables ou Tailwind dark:).
* Dados do usuário nunca saem do Firestore sem ação explícita do próprio usuário.
0.3 Como usar este PRD
1. Leia a seção 1 (Visão Geral) para entender o sistema atual.
2. Consulte a seção 2 (Arquitetura) antes de criar qualquer arquivo novo.
3. Para cada sprint, implemente apenas os itens da seção 4 correspondente.
4. Ao finalizar cada item, marque como concluído no backlog (seção 5).
5. Em caso de dúvida sobre escopo, consulte a seção 3 (Princípios de Design).
1. Visão Geral do Sistema
1.1 Propósito
O Fidúcia é um assistente financeiro pessoal de uso estritamente individual. Não é um produto comercial. Seu único objetivo é automatizar ao máximo o registro, a categorização e a análise das finanças do seu criador, reduzindo o atrito do controle financeiro ao mínimo possível.
1.2 Usuário
Usuário único e identificado. Não há multiusuário, compartilhamento de dados entre contas ou funcionalidades sociais. Todo o design de UX e toda a lógica de negócio devem ser otimizados para um único perfil de uso intenso e frequente.
1.3 Princípio norteador


Zero lançamento manual. Todo lançamento deve ser originado de uma captura automática (SMS, notificação, importação de arquivo) e exigir do usuário apenas uma confirmação rápida — no máximo 2 toques para registrar qualquer transação.
1.4 Estado atual (v1.0)
Funcionalidades já implementadas e estáveis — não alterar sem instrução:
* Autenticação Google Sign-In com isolamento de dados por uid.
* Dashboard com métricas de saldo, receitas, despesas e fatura de cartão.
* Extrato completo com filtros por conta, categoria, tipo e busca textual.
* Plano de Contas com categorias personalizadas e ícones temáticos.
* Contas (corrente, poupança, carteira) e Cartões de Crédito com limite e datas.
* Metas financeiras (Cofrinho) com aportes progressivos.
* Orçamentos mensais por categoria com alertas visuais progressivos.
* Importador OFX nativo com pré-visualização em tabela e categorização manual.
* Suporte nativo a modo claro/escuro.

	3. Princípios de Design e UX
3.1 Hierarquia de automação
Toda nova funcionalidade deve ser avaliada nesta ordem de preferência:
1. Automático silencioso: sistema age sem qualquer interação (ex: lançamento recorrente no dia certo).
2. Confirmação com 1 toque: sistema captura e apresenta, usuário apenas confirma (ex: SMS parseado).
3. Sugestão inteligente: sistema preenche campos, usuário ajusta se necessário (ex: categorização automática).
4. Assistido: formulário com campos pré-preenchidos baseados em histórico.
5. Manual puro: formulário em branco — usar apenas quando não há alternativa.
3.2 Dashboard como centro de controle
O Dashboard é a tela principal e deve responder a duas perguntas em menos de 3 segundos de leitura:
* Minha situação financeira agora está boa ou ruim?
* O que preciso fazer hoje? (confirmações pendentes, contas a vencer, metas atrasadas) .
3.3 Mobile-first
O sistema será acessado majoritariamente via celular. Toda nova tela deve ser desenvolvida primeiro para viewport 375px e depois adaptada para desktop. Botões de ação mínimo 44x44px. Formulários com campos grandes e teclado numérico ativado para valores.
3.4 Feedback imediato
Toda ação do usuário deve ter resposta visual em menos de 100ms (otimista UI — atualizar a interface antes da confirmação do Firestore, reverter em caso de erro).
4. Roadmap de Desenvolvimento
Sprint 1 — Importação Completa (Prioridade Máxima)


Objetivo: aceitar qualquer arquivo de extrato bancário sem configuração manual repetida.
4.1.1 Importador CSV/XLS com perfis salvos
* Contexto: atualmente só existe importador OFX. CSV e XLS são os formatos mais comuns de exportação bancária brasileira (Nubank, Itaú, BB, XP Investimentos).
* Requisitos funcionais:
   * Aceitar arquivos .csv, .xls e .xlsx via drag-and-drop ou seletor de arquivo.
   * Usar SheetJS (xlsx) para leitura de XLS/XLSX inteiramente no cliente (sem servidor).
   * Tela de mapeamento de colunas: usuário indica quais colunas são Data, Valor, Descrição, Tipo.
   * Detectar automaticamente separador decimal (vírgula ou ponto) e formato de data (DD/MM/YYYY, YYYY-MM-DD).
   * Salvar perfil de importação em users/{uid}/importProfiles com nome dado pelo usuário.
   * Na próxima importação do mesmo banco, carregar perfil salvo automaticamente.
   * Detectar duplicatas por hash antes de confirmar (campo hash do schema Transaction).
   * Exibir painel de revisão idêntico ao do OFX antes de confirmar a importação.
* Requisitos técnicos:
   * Componente: /src/components/Importer/CSVImporter.jsx.
   * Hook: /src/hooks/useImportProfile.js — CRUD de perfis no Firestore.
   * Utilitário: /src/utils/parseCSV.js — parsing + detecção de formato.
   * Utilitário: /src/utils/deduplication.js — geração e verificação de hash.
   * Sem dependências além de SheetJS (verificar se já está no projeto, senão: npm install xlsx).
4.1.2 Importador PDF de extrato
* Contexto: alguns bancos (especialmente corretoras e bancos tradicionais) só oferecem extrato em PDF.
* Requisitos funcionais:
   * Aceitar arquivos .pdf via drag-and-drop.
   * Usar pdfjs-dist para extração de texto no cliente.
   * Aplicar heurísticas de parsing: detectar linhas com padrão data + valor + descrição.
   * Apresentar resultado no mesmo painel de revisão do OFX/CSV antes de confirmar.
   * Exibir aviso quando a extração for parcial ou incerta (PDF escaneado sem OCR).
* Requisitos técnicos:
   * Componente: /src/components/Importer/PDFImporter.jsx.
   * Dependência: pdfjs-dist (verificar se já está no projeto).
   * Reutilizar painel de revisão já existente do importador OFX.
Sprint 2 — Captura Automática por SMS e Notificação


Objetivo: eliminar completamente o lançamento manual para gastos no cartão e conta corrente.
4.2.1 App Android companion para captura de notificações
* Contexto: bancos brasileiros enviam notificações push e SMS com padrão consistente. Esta é a peça central da automação. Solução recomendada: app Android simples (React Native / Expo) que escuta notificações e envia para o Fidúcia via Cloud Function.
* Arquitetura recomendada:
   * App Expo (React Native) com permissão de Notification Listener (Android NotificationListenerService).
   * Filtrar notificações por lista de apps bancários configurável (Nubank, Bradesco, Itaú, BB, C6, Inter, etc.).
   * Para cada notificação relevante: extrair texto, aplicar regex por banco, montar payload.
   * POST para Firebase Cloud Function endpoint autenticado (token do usuário Firebase).
   * Cloud Function valida token, cria Draft no Firestore users/{uid}/drafts.
   * App não armazena dados localmente — apenas encaminha.
* Regex de parsing por banco (implementar pelo menos):
   * Nubank: "Compra de R\$ X,XX em ESTABELECIMENTO".
   * Itaú: "Compra no cartão \*\_1234: R\$ X,XX em ESTABELECIMENTO".
   * Bradesco: "Compra aprovada: R\$ X,XX - ESTABELECIMENTO".
   * Banco do Brasil: "Compra: ESTABELECIMENTO R\$ X,XX".
   * Padrão genérico de fallback: detectar padrão R$ X,XX e capturar contexto.
* Requisitos da Cloud Function:
   * Endpoint: POST /api/sms-capture (Firebase Cloud Functions v2).
   * Autenticação: Bearer token Firebase Auth obrigatório.
   * Input: { rawText, bankApp, timestamp }.
   * Output: { draftId, parsedAmount, parsedDescription, confidence }.
   * Arquivo: /functions/src/smsCapture.ts.
4.2.2 Captura via e-mail (Gmail)
* Contexto: muitos bancos enviam comprovantes e alertas por e-mail. Google Apps Script ou Gmail API permite monitoramento automático.
* Requisitos funcionais:
   * Configurar trigger Google Apps Script que roda a cada 15 minutos.
   * Buscar e-mails não lidos com labels/remetentes configurados pelo usuário (ex: noreply@nubank.com.br).
   * Aplicar regex de parsing no corpo do e-mail.
   * Criar Draft no Firestore via Firebase Admin SDK ou REST API autenticada.
   * Marcar e-mail como lido após processamento.
* Configuração: Arquivo de configuração do usuário: users/{uid}/emailCapture (lista de remetentes + regex customizáveis).
Sprint 3 — Rascunhos e Fila de Confirmação


Objetivo: criar o ponto central de convergência de todas as capturas automáticas.
4.3.1 Tela de Rascunhos Pendentes
* Contexto: SMS, notificações e e-mails capturados automaticamente geram Drafts no Firestore. Esta tela é onde o usuário confirma, ajusta e descarta esses rascunhos.
* Requisitos funcionais:
   * Card de alerta no Dashboard: "X lançamentos aguardando confirmação" com link para a tela.
   * Lista de drafts ordenada por data, mais recentes primeiro.
   * Para cada draft: valor, estabelecimento/descrição, data, banco de origem, confiança da categorização.
   * Ação "Confirmar" com 1 toque: categoria sugerida aceita, draft vira Transaction.
   * Ação "Editar antes de confirmar": abre formulário pré-preenchido.
   * Ação "Descartar": remove o draft (com confirmação).
   * Confirmação em lote: selecionar todos e confirmar de uma vez.
   * Expiração automática: drafts com mais de 7 dias são marcados como expirados e removidos da fila.
* Requisitos técnicos:
   * Componente: /src/components/Drafts/DraftQueue.jsx.
   * Hook: /src/hooks/useDrafts.js — listener em tempo real na coleção drafts.
   * Animação de saída ao confirmar/descartar (CSS transition, não biblioteca).
Sprint 4 — Automação Inteligente de Categorização


Objetivo: eliminar a necessidade de categorizar manualmente qualquer lançamento.
4.4.1 Motor de regras de categorização
* Requisitos funcionais:
   * Interface em /configurações/regras para criar regras do tipo: SE descrição contém [texto] ENTÃO categoria = [X], conta = [Y].
   * Suporte a múltiplas condições por regra (AND).
   * Ordem de prioridade configurável entre regras.
   * Regras armazenadas em users/{uid}/categorizationRules.
   * Motor de regras executado no cliente antes de criar qualquer Draft ou Transaction.
   * Resultado inclui confidenceScore: 1.0 para match exato, menor para match parcial.
* Requisitos técnicos:
   * Utilitário: /src/utils/categorizationEngine.js — função pura, testável.
   * Deve funcionar offline (sem chamada de rede).
   * Testes obrigatórios: /src/utils/__tests__/categorizationEngine.test.js.
4.4.2 Aprendizado por frequência
* Requisitos funcionais:
   * Após o usuário categorizar manualmente um lançamento, verificar se a mesma descrição (normalizada) já foi categorizada antes.
   * Se 3 ou mais ocorrências com mesma categoria: criar regra automática com confirmação do usuário ("Sempre categorizar UBER como Transporte?").
   * Normalização de descrição: uppercase, remover números de transação, remover espaços extras.
   * Sugestão aparece como toast não-intrusivo, não como modal bloqueante.
Sprint 5 — Lançamentos Recorrentes


Objetivo: contas fixas (aluguel, assinaturas, mensalidades) jamais precisam ser lançadas mais de uma vez.
4.5.1 Série de recorrências
* Requisitos funcionais:
   * Ao criar ou editar um lançamento, opção "Tornar recorrente" abre configuração de frequência.
   * Frequências: diária, semanal, quinzenal, mensal (dia fixo ou último dia do mês), anual.
   * Armazenar série em users/{uid}/recurrences com próxima data de vencimento.
   * Cloud Function agendada (Firebase Scheduled Function, todo dia às 08h) gera Drafts para recorrências do dia.
   * Draft gerado aparece na fila de confirmação com 1 toque para confirmar.
   * Opção de confirmar automaticamente sem passar pela fila (para contas com valor fixo e conta bancária definida).
   * Edição de série: alterar apenas ocorrência atual ou todas as futuras.
   * Encerramento de série: definir data de término ou encerrar manualmente.
* Requisitos técnicos:
   * Arquivo: /functions/src/generateRecurrences.ts (Firebase Scheduled Function).
   * Componente: /src/components/Recurrences/RecurrenceManager.jsx.
   * Hook: /src/hooks/useRecurrences.js.
Sprint 6 — Dashboard Aprimorado


Objetivo: responder em 3 segundos "estou bem financeiramente hoje?".
4.6.1 Novos indicadores do Dashboard
* Cards a adicionar (manter layout de grid responsivo existente):
   * "Quanto posso gastar hoje?" — saldo disponível dividido pelos dias restantes do mês, descontando recorrências previstas.

$$Fórmula: \frac{\text{saldo\_atual} - \text{soma\_recorrencias\_restantes\_no\_mes}}{\text{dias\_restantes}}$$

   * "Saúde financeira do mês" — índice 0-100. Cálculo: 40% orçamento geral utilizado + 30% metas no prazo + 30% saldo positivo vs mês anterior.
   * "Próximas obrigações" — lista de recorrências e faturas dos próximos 7 dias.
   * Variação % vs mês anterior em todos os cards de métricas existentes (seta verde/vermelha).
   * Gráfico de projeção de saldo:
   * Gráfico de linha: saldo real dos últimos 30 dias + projeção pontilhada dos próximos 30.
   * Projeção baseada na média de receitas e despesas dos últimos 3 meses.
   * Linha pontilhada para projeção, linha sólida para realizado.
   * Biblioteca: Recharts (verificar se já está no projeto).
4.6.2 Análises avançadas (tela separada)
   * Nova rota /analises com:
   * Mapa de calor de gastos: calendário mensal com intensidade de cor por dia (maior gasto = cor mais escura). Biblioteca: react-calendar-heatmap ou implementação própria com SVG.
   * Top 10 estabelecimentos do mês: ranking por valor total gasto.
   * Tendência por categoria: mini gráfico sparkline (últimos 6 meses) para cada categoria ativa.
   * Evolução patrimonial: gráfico de área com soma de todos os saldos mês a mês.
Sprint 7 — PWA e Notificações Push


Objetivo: experiência de app nativo no celular sem instalar nada da loja.
4.7.1 Progressive Web App
   * Requisitos:
   * Configurar vite-plugin-pwa com manifest completo (nome, ícones, cores, display: standalone).
   * Service Worker com estratégia cache-first para assets estáticos.
   * Firestore offline persistence já habilitada — verificar se enableIndexedDbPersistence está ativo.
   * Tela de "sem conexão" amigável quando offline e sem cache disponível.
   * Ícone do app na tela inicial iOS e Android.
4.7.2 Notificações push via Firebase Cloud Messaging
   * Alertas a implementar:
   * Novo rascunho capturado (SMS/e-mail): "Nova transação de R$X em [estabelecimento]".
   * Orçamento de categoria atingindo 80%: "Atenção: 80% do orçamento de Alimentação utilizado".
   * Fatura a vencer em 3 dias: "Fatura [Cartão] vence em 3 dias — R$X".
   * Recorrência gerada aguardando confirmação.
   * Meta atingida: "Parabéns! Meta [nome] concluída!".
Sprint 8 — Funcionalidades de Qualidade de Vida


Objetivo: pequenas adições de alto impacto no dia a dia.
4.8.1 Split de lançamentos
Dividir um único lançamento em múltiplas categorias. Caso de uso clássico: compra no mercado com produtos de diferentes categorias.
   * Requisitos:
   * Botão "Dividir" em qualquer Transaction no extrato.
   * Interface para adicionar N linhas de divisão (descrição + valor + categoria).
   * Validação: soma das divisões deve ser igual ao valor total original.
   * Lançamento original recebe status "split_parent", divisões são Transactions independentes com referência ao pai.
4.8.2 Entrada rápida por linguagem natural
   * Requisitos:
   * Campo de texto no topo do Dashboard ou atalho de teclado "N" (desktop).
   * Parser de linguagem natural: "35 almoço", "paguei 150 conta de luz", "recebi 2000 freelance".
   * Regras de parsing em /src/utils/naturalLanguageParser.js (sem IA externa, apenas regex + heurísticas).
   * Formulário pré-preenchido com resultado do parsing para revisão antes de salvar.
4.8.3 Exportação e backup
   * Requisitos:
   * Exportar extrato filtrado em CSV (qualquer período, qualquer filtro ativo).
   * Exportar backup completo em JSON (todas as coleções do usuário).
   * Botões na tela de Configurações, sem confirmação desnecessária.
   * CSV com cabeçalho: Data, Descrição, Valor, Tipo, Categoria, Conta, Tags.

	6. Requisitos de Segurança
Regras invioláveis — qualquer implementação que viole estas regras deve ser recusada e reportada:
   1. Regras Firestore: Toda leitura e escrita deve passar pela regra request.auth.uid == resource.data.uid. Nunca criar documentos sem o campo uid.
   2. Cloud Functions: Toda Cloud Function que recebe dados do usuário deve validar o token Firebase Auth antes de qualquer operação.
   3. Dados locais: Nenhum dado financeiro deve ser armazenado em localStorage, sessionStorage ou IndexedDB não gerenciado pelo Firestore SDK.
   4. SMS/Notificações: O app companion Android não armazena dados — apenas encaminha. A Cloud Function é o único ponto de persistência.
   5. Exportação: A função de exportação deve ser chamada apenas autenticada e nunca expor dados de outros usuários.
   6. Dependências: Qualquer nova dependência npm deve ser auditada com npm audit antes de ser adicionada.

	Lançamento capturado automaticamente aguardando confirmação do usuário.

	Lançamento financeiro confirmado e persistido no extrato.

	Número 0.0 a 1.0 indicando confiança da categorização automática.

	Divisão de um único lançamento em múltiplas categorias.

	Progressive Web App — sistema instalável no celular via navegador.
	FCM
	Firebase Cloud Messaging — serviço de notificações push do Google.
